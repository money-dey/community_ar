# Community AR Phase 1 — Perception

The perception layer. Every frame produces a complete `PerceptionFrame`:
landmarks, iris, blendshapes, face pose, hair mask, skin tone — all
temporally smoothed and ready to drive effects.

By the end of Phase 1, you can see landmark dots tracking your face,
hair tinted green by the mask, and live performance numbers. **Until you
see those overlays drawn correctly, nothing in Phase 2+ can be trusted.**

## What's new vs Phase 0

| Component | Purpose |
|---|---|
| `NeuralBackend` | Abstract interface for ML inference (TFLite/Core ML) |
| `FaceLandmarker` | 468 face landmarks + 52 blendshapes (MediaPipe FaceMesh) |
| `IrisLandmarker` | Iris center + 5 contour points per eye (MediaPipe Iris) |
| `HairSegmenter` | Per-pixel hair mask |
| `SelfieSegmenter` | Person-vs-background mask (for BackgroundEffect later) |
| `OneEuroFilter` | Adaptive temporal smoothing — *single most important quality lever* |
| `solveFacePose` | PnP-based 6DoF face pose against a canonical 3D model |
| `SkinToneEstimator` | Trimmed-mean skin baseline luma (every ~5 frames) |
| `PerceptionPipeline` | Orchestrator |
| `DebugOverlay` | Renders landmarks/masks on top of camera for verification |

## Architectural notes

**Perception-on-demand.** `PerceptionPipeline::setRequirements()` accepts a
`PerceptionInputs` struct that is the **union** of all enabled effects'
needs. Models are loaded lazily on first request and skipped entirely when
not needed. If no enabled effect needs iris landmarks, iris inference never
runs.

**Temporal smoothing is non-negotiable.** Raw model output has sub-pixel
jitter on every landmark every frame. Without One-Euro filtering, the
entire pipeline looks "alive" in a bad way. The defaults
(minCutoff=1.0, beta=0.007) are calibrated for 30fps face landmarks; the
debug UI lets you tune them live.

**Skin tone is decoupled from the per-frame critical path.** It runs every
5 frames on the GPU asynchronously, with results read back via PBO/MTLBuffer.
Worst-case staleness is ~170ms at 30fps, which is invisible to users since
real skin tone changes far slower than that.

**PnP without dependencies.** The face pose solver is ~200 lines of inline
numerical code (Rodrigues, Gauss-Newton on reprojection error, Cholesky
solve). Empirically converges in 3-5 iterations to sub-pixel reprojection
error for stable face geometry. No Eigen or OpenCV needed.

## Verification checklist

Before declaring Phase 1 done:

### Android
- [ ] App launches, camera works (Phase 0 still passing)
- [ ] Landmark dots appear and track face movement smoothly
- [ ] Dots don't jitter visibly when face is stationary
- [ ] Dots respond quickly to fast head turns (no "lag/swimming")
- [ ] Iris circles appear over each iris and track during gaze movement
- [ ] Hair mask tints hair green; works on diverse hair textures
- [ ] Stats panel shows reasonable times: FaceMesh < 8ms, Iris < 4ms,
      Hair < 10ms on mid-range device
- [ ] Adjusting minCutoff slider visibly affects smoothness
- [ ] Adjusting beta slider visibly affects motion responsiveness
- [ ] Skin baseline luma updates ~6 times/second when face is visible

### iOS
- [ ] Same as Android
- [ ] Core ML accelerator shows "ANE" in log on supported devices (A12+)
- [ ] Inference times are 2-3× faster than Android equivalents (ANE benefit)

### Cross-platform
- [ ] **Critical: tested on diverse skin tones.** The skin tone estimator's
      trimmed-mean output should produce sensible values across light, medium,
      and dark skin. Document any biases observed.
- [ ] **Critical: tested on diverse hair textures.** Hair segmentation
      should work on straight, wavy, curly, coily, locs, braids. Note any
      failure cases — this is exactly the kind of issue Community AR exists
      to address.

## Known shortcuts (clean up before Phase 2)

- TFLite GPU delegate texture binding is stubbed; current path uses CPU
  upload for the camera frame. Real perf gain comes from binding the OES
  texture directly. Fix before measuring real frame times.
- Iris model right-eye pass omitted from the scaffold; only left eye is
  populated. Mirror the pattern.
- Skin tone GPU compute pass is documented in the shader but not yet bound
  to compute primitives in RenderContext.
- Multi-face support: model output assumed to be one face. Real MediaPipe
  models emit a face count tensor we need to parse.
- The debug overlay uses one draw call per landmark dot (468 dots = 468 draws).
  Acceptable for debug, not for production. Switch to instanced rendering
  if we ever want this in shipping builds.
